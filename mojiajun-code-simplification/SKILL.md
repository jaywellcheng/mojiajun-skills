# 代码简化铁律

> "新成员读这段代码会不会比原版更快理解？"——简化唯一标准

## 何时做/不做

| 做 | 不做 |
|----|------|
| 功能能跑但太重 | 代码已够简洁 |
| 嵌套3层以上 | 你不完全理解这段代码 |
| 函数超50行 | 性能敏感且简化会明显变慢 |
| 命名模糊（data/result/temp） | 即将重写整个模块 |

## 五条核心原则

### 1. 行为完全不变
改动只能改表达方式。所有输入输出、副作用、边界情况必须一致。**每次改完跑测试，测试不改。**

### 2. 遵循项目惯例
简化 = 让代码更符合现有代码库风格，不是引入你的偏好。import方式、命名、错误处理全都跟项目走。

### 3. 清晰胜过聪明
```python
# ❌ 聪明但不清晰
label = "新" if is_new else "更新" if is_updated else "活跃"

# ✅ 清晰
def get_label(item):
    if item.is_new:    return "新"
    if item.is_updated: return "更新"
    return "活跃"
```

### 4. 别过度简化
- 别把命名清晰的helper内联掉
- 别把两个简单函数合成一个复杂函数
- 别为了省行数牺牲可读性（少行 ≠ 好代码）

### 5. 只改刚改过的
默认只简化最近改动过的代码。不相关的drive-by重构 = 噪声diff + 回归风险。

## 简化流程

### Step 1: Chesterton's Fence（先理解再动手）
> 看到路上有一道栅栏，不理解为什么在那里——别拆。先搞清楚原因，再判断原因是否仍然成立。

**动手前回答：**
- 这段代码的职责是什么？谁调它/它调谁？
- 边界情况和错误路径是什么？
- 为什么当初写成这样？（性能？平台限制？历史原因？）
- `git blame` 看看原始上下文

答不上来 → 不够格简化。

### Step 2: 找简化机会

| 信号 | 做法 |
|------|------|
| 嵌套3层+ | 卫语句(early return)或拆helper |
| 函数超50行 | 拆成多个职责单一的函数 |
| 嵌套三元 | if/elif 替代 |
| `data/result/temp` | 改成 `user_profile/validation_errors` |
| 缩写 `usr/cfg` | 全拼（除非通用如 id/url/api） |
| 注释解释"是什么" | 删注释（代码已足够清晰） |
| 注释解释"为什么" | **保留！**这是意图 |
| 重复代码5行+ | 提取共享函数 |
| 死代码 | 确认后删除 |
| 过度设计（工厂的工厂） | 用最简单的直接方式 |

### Step 3: 渐进式修改
**一次只做一个简化，每次改完跑测试。** 重构和功能改动必须分开提交。
