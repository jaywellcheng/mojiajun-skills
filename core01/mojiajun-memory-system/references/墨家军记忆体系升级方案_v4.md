## 二、架构哲学：为什么选择算子级递归 + 文档化编排

### 2.1 引擎 vs 模型：谁管记忆

这是我们与LCM论文共同的核心立场：

```
❌ RLM方式（Reactive Language Model）：
   模型自己写代码管理自己的上下文 — 灵活但不可靠。
   压缩可能越压越长（compaction failure），递归可能无限循环。

✅ LCM方式（引擎提供确定性算子）：
   模型只做决策（"这条要压缩"），引擎负责执行（"压缩到512 token"）。
   受控、可预测、数学上保证收敛。
```

**为什么这个选择对墨家军尤其重要——**
墨家军已经是多Agent系统（9个Agent + 子代理）。如果每个Agent都用自己的方式管理记忆，复杂度是O(n²)的不可控叠加。引擎统一管理上下文，每个Agent只做自己的事，复杂度是O(n × 1)。

```
Agent → 只做决策（"需要查这条记忆"）
引擎 → 执行展开、压缩、校验、存储
```

### 2.2 文档化编排：Karpathy 的 program.md 范式

autoresearch（71K⭐）证明了一件事：**最好的智能体编排可能就是没有编排框架**。一份写得好的自然语言指令，在足够强的 LLM 面前，可以替代大量的工程基础设施。

```
传统方式：LangGraph / CrewAI / AutoGen — 状态机、有向图、工具注册表
Karpathy方式：一个 Markdown 文件 — 告诉 Agent 怎么思考、优先试什么、循环不要停
```

对墨家军的启示：我们不仅需要分层存储**信息**（memory），还需要一个独立的**行为纲领层**（program）——告诉每个 Agent **在什么场景下用什么策略做决策**。这是比 memory 更高一层的指令。

## 四、改造点详解

### 4.1 四层分级加载

在 CLAUDE.md 三层级基础上，增加行为纲领层：

| 层级 | 位置 | 内容 | 加载方式 | 对标 |
|:----:|------|------|---------|------|
| **行为纲领层** | 项目根目录 `program.md` | Agent 行为指令：怎么思考、优先试什么、循环不要停 | 每次会话全量 | autoresearch 的 `program.md` |
| **用户级** | `~/.hermes/memory/` | 小云的身份、大威偏好、通用规则 | 每次会话全量 | `~/.claude/CLAUDE.md` |
| **项目级** | 项目根目录下的memory配置 | 项目结构、操作协议、命名规范 | 每次会话全量 | `./CLAUDE.md` |
| **子级** | 子目录下的memory配置 | 特定场景规则（窑滚人生/天青浅各一套） | **按需加载** | 子目录`CLAUDE.md` |

**行为纲领层的内容示例（墨蓝的 program.md）：**
```
拿到标题任务 → 先查标题库 → 出3个候选 → 用5问自检 → 挑最优
→ 不满足就重写，不要停下来问人类
→ 标题至少触发2个心理开关，做不到就换公式
→ 每完成一篇笔记，自动记录用了哪个公式、触发了哪些开关
→ 如果连续3篇笔记互动率低于基线，主动检查是否需要调整策略
```

**关键改动：**
- 行为纲领层和用户级全量加载；子级内容在 Agent 触及该目录时才触发加载。

### 4.2 200行截断 + 系统提示瘦身

**现在的MEMORY.md**：39行 / 3,859字符，未触及上限但需提前架构。

**改造方案：**
- MEMORY.md 维持前200行或25KB（以先到者为准）加载到活动上下文
- 超出部分存入不可变存储，通过`lcm_expand`按需展开
- 旧的/不活跃的记忆自动降级到 Level 2/Level 3，不占启动token

**对接现有 memdir 系统：**
- 超出截断的记忆条目自动从 memdir 的 INDEX.md 中按需加载，不重复存储
- memdir 的独立文件作为原始节点保留在不可变存储中，摘要节点指向 memdir 索引

**系统提示瘦身（对标 ECC 的 /system-prompt-lean）：**
- 每次 API 调用都有一份固定的"系统提示"开销（工具定义、角色指令等）
- 定期审查系统提示中是否有多余的工具 Schema、过时的角色描述
- 高频工具保持完整 Schema，低频工具降级为一行摘要
- 目标：系统提示控制在当前大小的 70% 以内

### 4.3 大文件探索摘要（超过25K token的处理）

LCM 对超过25K token的文件不直接加载，而是生成**类型感知的探索摘要**：

| 文件类型 | 探索摘要内容 |
|---------|-------------|
| SQL/数据库文件 | 表结构概览、索引、外键关系 |
| JSON/YAML | 键路径树、顶层结构 |
| 代码文件 | 函数签名列表、类层次结构、公共接口 |
| 非结构化文本 | LLM 生成的内容摘要 |

Agent 看到的是结构骨架（<1K token），需要时才通过`lcm_expand`展开具体内容。

**改造方案：**
- 新增文件token阈值检测（默认25K token）
- 超限文件自动转为探索摘要，不加载原文
- 探索摘要作为 Level 2 节点持久化到 DAG

### 4.4 Skill 按需加载（取代全量加载）

**现在的Skill**：调用`skill_view()`时加载完整内容。

**改造方案：**
- skill 的简短描述（名字+一句话说明）默认加载到活动上下文（类似 skills_list）
- 完整的 skill 内容只在 Agent 匹配到需要使用时才通过`skill_view()`加载
- 高频 skill（被调用≥3次）自动升级到 Level 1 摘要，保持在活动上下文中
- 低频 skill 保持在 Level 3（仅描述），节省 token

### 4.5 四级压缩协议 + Schema校验

在 LCM 三级升级基础上，增加调用级微压缩（Level 0）：

| 层级 | 触发条件 | 内容 | token消耗 | 校验机制 | 来源 |
|:----:|---------|------|:---------:|---------|------|
| **Level 0** | 每次 API 调用前 | 自动清理冗余消息（重复文件读取、噪声数据） | 几乎为零 | — | ECC 的 microcompact |
| **Level 1** | 消息量≥8K token | LLM 生成保留细节的摘要 | 完整 | JSON Schema 校验输出格式 | LCM |
| **Level 2** | Level 1 后继续增长 | LLM 生成要点式摘要，目标减半 | 精简 | JSON Schema 校验，注入纠错后重试 | LCM |
| **Level 3** | Level 2 后继续增长 | 确定性截断到 512 token | 固定 | **不调用 LLM，纯确定性操作** | LCM |

> **关键保证：** Level 3 不含 LLM 调用——纯确定性截断到 512 token。
> **数学上保证每次压缩必然产出更短的上下文，永不出现"压缩失败"（compaction failure）。**
> 这是 LCM 相比 RLM 最硬核的优势——不用 LLM 的环节，就是 100% 可靠的环节。

**Schema校验流程：**
```
LLM输出 → JSON Schema校验
  ├─ 通过 → 写入DAG
  └─ 不通过 → 注入纠错消息 → LLM重试（最多K次）
                └─ 仍不通过 → 降至Level 3确定性截断
```

### 4.6 LLM-Map：并行处理引擎

这是 LCM 打赢 Claude Code 的核心杀招之一：

```
Input (JSONL数组)
  │
  ▼
┌─────────────────────────────────────┐
│        Worker Pool（16路并发）       │
│  ┌──────┐ ┌──────┐     ┌──────┐    │
│  │ w1   │ │ w2   │ ... │ w16  │    │
│  └──┬───┘ └──┬───┘     └──┬───┘    │
│     │        │            │        │
│ 每条独立调LLM + Schema校验         │
│ 失败→自动重试（最多K次）            │
└─────────────────────────────────────┘
  │
  ▼
结果写回LCM Store（不可变存储）
  │
  ▼
返回Summary Handle给Agent
```

**适用场景：**
- 批量分类/提取/评分（无副作用的纯函数任务） → 用 `llm_map`
- 多条记忆同时压缩/升级
- 大量小文件的探索摘要生成
- **多条创作任务并行**（如"同时写3篇不同主题的笔记，每篇都是完整创作会话"） → 用 `agentic_map`

**与 delegate_task 的区别：**
| | LLM-Map | delegate_task |
|---|---------|---------------|
| 任务类型 | 纯函数 LLM 调用 | 多步推理+工具调用 |
| 并发模型 | 16路 Worker Pool | 主-子代理会话 |
| 输出 | 结构化数据（Schema校验） | 自由文本+文件 |
| 重试策略 | 引擎自动重试 | 子代理自行处理 |

### 4.7 零成本连续性 + 手动 compact

```
低于 8K token（τ_soft）：零开销，完全不压缩
8K ~ 32K token：后台异步压缩，用户无感知
超过 32K token（τ_hard）：阻塞压缩
```

**手动 compact 最佳实践（对标 ECC）：**
> 不要等自动触发。上下文超过 100K token 时手动执行 `/compact`。
> 自动触发阈值是 contextWindow - 13,000，到那时已经花了不少冤枉钱了。

### 4.8 会话钩子 + SQLite 持久化（对标 ECC Hook 系统）

**痛点：** 早上跟 Agent 讨论了半小时的架构方案，下午再打开——上下文没了，重来一遍。Token 白烧。

**方案：**
```
session_start hook → 从 SQLite 加载上次会话的 Summary Handle
  → 展开到活动上下文（Level 1 摘要）
  → Agent 看到的不再是"空白"

session_end hook → 将当前上下文生成 Summary
  → 写入 SQLite（键：project_name + session_id）
  → 下次 session_start 自动恢复
```

- 存储：`~/.hermes/session_context.db`（SQLite，表 `session_summaries`）
- 字段：project_name, session_id, summary_level, dag_node_id, created_at
- 过期策略：保留最近 30 天，超过的自动降级到 Level 3 摘要

### 4.10 安全扫描集成（对标 ECC AgentShield）

**现状：** 我们有 `requesting-code-review` skill，但它需要手动触发。缺乏每次代码变更的自动安全审查。

**方案：**
- 在 session_end hook 中增加安全扫描步骤
- 对接现有 `requesting-code-review` skill 的 102 条规则（静态安全扫描部分）
- 扫描结果写入 `system_change_log`，标记 `security_scan: passed/failed`
- 失败时在下次 session_start 主动提醒

```
session_end hook：
  1. 生成上下文摘要 → 写入 SQLite
  2. git diff --cached | 静态安全扫描（secret/shell injection/SQL injection/eval）
  3. 扫描结果 → system_change_log（security_scan 字段）
  4. 不通过 → 下次 session_start 主动提醒
```

### 4.11 持续学习 + 置信度评分（对标 ECC instinct-status）

**现状：** 五步自进化闭环有 Skill 触发（高频 insight → Skill 更新建议），但触发是**被动**的（等 insight 命中 ≥3 次）。

**方案：** 增加主动提取机制：
```
每 10 次 Agent 任务后 → 自动扫描对话历史
  → 提取可复用的操作模式（如"墨蓝写笔记时总是先查标题库"）
  → 生成「技能置信度评分」（0-100）
  → ≥70 分 → 自动建议升级为 Skill 或写入行为纲领层
  → 30-69 分 → 标记为候选，等待更多数据
  → <30 分 → 丢弃
```

> 对接现有五步自进化闭环：置信度评分 ≥70 的行为模式 → 直接写入 Skill 触发队列，无需等 3 次命中。

### 4.12 记忆管理原则

1. **200行是上限** — 超过就拆到子级或降级
2. **删掉这条会犯错吗？** — 不会就删
3. **高频放上层，低频放下层** — 每次都要的才放用户级
4. **定期重构** — 每月过一遍，删过时规则
5. **HTML注释不占token** — 备注用`<!-- -->`包住
6. **简洁性优先** — 增加复杂性的改进必须有 2 倍以上收益才保留；删掉不用的代码/规则 = 永久胜利（对标 autoresearch 的简洁性偏好）

## 八、关键结论

> **墨家军已有的五步自进化闭环（taste filter → 验证区 → Eval → 遗忘 → 触发）解决了「记什么不记什么」的问题。**
>
> **四层分级加载（行为纲领 + 用户级 + 项目级 + 子级）+ 200行截断，解决了「怎么省context window」的问题。**
>
> **四级DAG摘要（Level 0微压缩 + 三级升级）+ Schema校验 + 数学收敛保证，解决了「记太多塞不下且不会越压越膨胀」的问题。**
>
> **LLM-Map并行引擎 + 算子级递归，解决了「多Agent各自管理记忆导致复杂度爆炸」的问题。**
>
> **会话钩子 + SQLite持久化，解决了「每次从零开始，上下文说丢就丢」的问题。**
>
> **持续学习 + 置信度评分，解决了「经验无法沉淀为能力」的问题。**
>
> **行为纲领层（program.md），解决了「Agent知道有什么信息但不知道怎么决策」的问题。**
>
> **安全扫描集成，解决了「代码变更无人审查」的问题。**
>
> **八件事加起来 = 完整的墨家军记忆体系。大部分基础设施已就位，缺的是「组织方式」、「加载策略」、「并行引擎」和「自动触发机制」。**
