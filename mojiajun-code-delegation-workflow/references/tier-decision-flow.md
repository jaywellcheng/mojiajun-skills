# 三级信任决策树

## 概述

墨家军代码分派工作流 v2 引入三级信任机制，根据任务风险自动选择审核和部署策略。
决策树由小川在执行阶段1（分析）时遍历。

---

## 各级详解

### Tier 1 — 严格审核

```
触发条件（任一满足即走 Tier 1）：
  - 触及 CRITICAL_PATHS
  - 变更 >150 行
  - 核心逻辑改动（调度器、认证、数据模型）
  - 新模块/新文件
  - AS 信誉未建立（前5次任务）
  - 非预置模板任务且变更 >50 行

流程：
  小川分析 → 分派AS → 墨码完整审核 → 小川逐行复核 → 小川手动部署

墨码要求：
  - 七条铁律逐项检查
  - 每项给出 pass/fail + 具体行号
  - 整体 verdict: PASS / FAIL

小川动作：
  - 阅读完整 diff
  - 逐行审查墨码标记的问题
  - 确认无遗漏后合并
  - 手动执行部署

风险控制：
  - 最高安全级别
  - 不允许任何自动跳过
```

### Tier 2 — 轻量审核

```
触发条件（全部满足）：
  - 不触及 CRITICAL_PATHS
  - 变更 ≤150 行
  - 非核心逻辑改动
  - 任务类型为修复Bug或小范围重构

流程：
  小川分析 → 分派AS（含 self-review 要求）→ 墨码快速审核 → 小川抽样复核 → 小川手动部署

墨码要求：
  - 仅检查严重(severity=high)和高危(severity=critical)问题
  - 跳过 style、docstring、naming 类 minor 问题
  - 输出：verdict + 仅列出 high/critical 问题

小川抽样规则：
  - 至少检查 diff 中 30% 的行
  - 重点区域：异常处理、边界条件、安全相关代码
  - 如发现任何问题 → 升级为逐行复核

AS 额外要求：
  - 必须附 self-review 注释（在代码中用 "# SELF-REVIEW:" 标记关键决策）
  - self-review 需覆盖：为什么这样写、替代方案、潜在风险

风险控制：
  - 中等安全级别
  - 抽样发现1个问题即升级 Tier 1
```

### Tier 3 — 自动部署

```
触发条件（全部满足）：
  - 墨码 verdict == "PASS"
  - 墨码 minor_count == 0
  - 变更 ≤50 行
  - 不触及 CRITICAL_PATHS
  - 任务来自预置模板（templates/）
  - AS 信誉良好（连续≥3次 Tier 2 通过，且已完成冷启动期）

流程：
  小川分析 → 选择预置模板 → 分派AS → 墨码审核 → auto_deploy 自动部署 → 通知小川

auto_deploy 自动判断逻辑：
  (见 scripts/auto_deploy.py)
  1. 验证墨码 PASS + minor_count=0
  2. 验证 lines_changed ≤ 50
  3. 验证 changed_files 不触及 CRITICAL_PATHS
  4. 全部通过 → cat管道传文件到目标主机
  5. systemd 文件：仅传输，不重启（需小川确认）

小川动作：
  - 无需复核（信任机制）
  - 收到通知后抽查（可选）
  - systemd 变更需手动确认重启

风险控制：
  - 多重条件过滤（模板 + 墨码 + 行数 + 路径 + 信誉）
  - 出错回滚路径：Git revert + 手动修复
  - systemd 永远不自动重启
```

## 关键路径（CRITICAL_PATHS）完整清单

| 路径 | 原因 | 触及时强制 Tier |
|------|------|:---:|
| `systemd/` | systemd unit 文件，错误可致服务不可用 | 1 |
| `.env` | 环境变量，包含数据库连接等敏感信息 | 1 |
| `auth.json` | 认证配置，错误可致安全漏洞 | 1 |
| `nginx/` | Nginx 配置，错误可致服务不可达 | 1 |
| `module_dispatcher.py` | 核心调度器，影响所有任务分发 | 1 |
| `task_queue schema` | 任务队列数据结构，影响持久化 | 1 |
| `ai-code-agent-governance/` | 治理规则，元级别的修改需最高审查 | 1 |
| `*secret*` / `*token*` / `*password*` | 凭据相关文件 | 1 |

---

## 冷启动期规则

新 AS（或新接入的代码代理）需经历冷启动期：

- **前 5 次任务**：强制 Tier 1，无论其他条件如何
- **第 6-10 次**：可进入 Tier 2，但需小川额外确认
- **第 11 次起**：正常使用 Tier 决策树
- **信誉重置条件**：连续 2 次 Tier 1 墨码 FAIL → 冷启动期重置

---

## 并行分派与 Tier 的交互

| 场景 | Tier 处理 | 说明 |
|------|-----------|------|
| 两个独立 Tier 2 任务 | 各自独立判断，可并行 | 墨码审核各自进行 |
| 一个 Tier 1 + 一个 Tier 3 | 并行分派，但 Tier 1 拉长整体耗时 | Tier 3 先完成也要等 Tier 1 |
| 两个 Tier 3 任务 | 理想场景，并行+自动部署 | 小川几乎零参与 |

---

## 降级/升级规则

### 自动升级（提升安全级别）

- Tier 2 墨码发现 high/critical 问题 → 升级为 Tier 1 逐行复核
- Tier 3 auto_deploy 判断不通过 → 降级处理
- 小川抽样发现任何问题 → 升级为逐行复核

### 降级路径（Tier 3 失败时）

```
Tier 3 自动部署被拒
      │
      ├── 墨码 FAIL → Tier 1（严格审核 + 重写/手动修）
      ├── 触及 CRITICAL_PATHS → Tier 1
      ├── minor_count > 0 → Tier 2（小川手动部署）
      └── lines_changed > 50 → Tier 2（小川手动部署）
```
